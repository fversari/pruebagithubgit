Bloque 1

Checklist de diagnóstico

Antes de construir nada, necesitás saber exactamente dónde está parado tu producto. El método es simple y brutal: abrir la URL en una pestaña de incógnito y usarla como si fueras el cliente. No el desarrollador. El cliente.

Evaluá cada criterio con honestidad: ✓ está / ✗ no está / parcial. Al final, identificá los dos problemas más críticos para resolver hoy.

  • 1
    El flujo principal funciona sin errores de consola
    Podés hacer la acción central de la app de principio a fin sin que nada se rompa visualmente ni aparezcan errores técnicos. El usuario nunca ve mensajes de error inesperados.
    Cómo verificarlo: Abrir DevTools → pestaña Console. Hacer el flujo completo. ¿Hay errores en rojo? Si hay, el MVP no está listo.
  • 2
    Los datos se guardan y se recuperan correctamente de Supabase
    Cuando el usuario crea un registro, ese dato persiste aunque cierre el navegador. Cuando vuelve a abrir la app, los datos están exactamente como los dejó. La base de datos no es decorativa.
    Cómo verificarlo: Crear un dato → cerrar la pestaña → reabrir la URL → verificar que el dato aparece.
  • 3
    La UI no se rompe en diferentes tamaños de pantalla
    Tanto en escritorio como en el celular, los elementos se ven correctamente. Nada se superpone, ningún texto desborda su contenedor, los botones son tocables en mobile.
    Cómo verificarlo: Redimensionar la ventana del navegador hasta ~375px de ancho, o abrir en el celular.
  • 4
    Los estados vacíos están manejados
    Cuando no hay datos — primer acceso, tabla vacía, búsqueda sin resultados — el usuario ve un mensaje claro que le explica qué hacer. No una pantalla en blanco ni una lista vacía sin contexto.
    Cómo verificarlo: Borrar todos los registros de la tabla principal y ver qué muestra la interfaz.
  • 5
    Se puede usar sin explicación previa
    Una persona que nunca vio tu app puede completar el flujo principal sin que le expliques nada. Si necesita que le cuentes qué significa cada botón o campo, el producto todavía no está listo para un cliente real.
    Cómo verificarlo: Pedirle a alguien que nunca vio la app que la use. Observar sin intervenir. Cada vez que preguntan "¿qué hace esto?", es un gap de UX.
  • 6
    El texto es claro — el usuario sabe qué hacer en cada pantalla
    Los labels, botones, mensajes de error y placeholders usan el vocabulario del dominio real, no texto genérico. El usuario nunca se pregunta "¿qué se supone que hago acá?"
    Cómo verificarlo: Leer todos los textos de la interfaz. ¿Hay "Enter text here", "Button", "My App"? ¿Los mensajes de error explican qué pasó y qué hacer?

"El diagnóstico honesto es la diferencia entre construir lo correcto y construir más de lo mismo que no funciona."

Bloque 2

Qué mejorar primero

Tenés 50 minutos de build. La tentación es atacar todo a la vez — no lo hagas. Un cambio, una prueba, confirmar que funciona. Recién ahí el siguiente. Si intentás refactorear todo de una vez, vas a terminar con el proyecto roto y sin tiempo.

Orden de prioridad estricto

Crítico
Flujo principal sin errores. Si el flujo central se rompe, todo lo demás es irrelevante.
Alta
Datos que persisten en Supabase. Si la base de datos no funciona, la app no tiene valor.
Alta
Estados vacíos y de error manejados. El usuario no puede quedar en blanco.
Media
Copy y textos del dominio real. Sin placeholders genéricos ni texto de v0 sin personalizar.
Media
Responsive básico. Que no se rompa en el celular.
Después
Design polish. Colores, tipografía, animaciones. Solo si todo lo de arriba está resuelto.

Problemas comunes y cómo resolverlos

🔐
Supabase RLS bloqueando lecturas o escrituras
La app no muestra ni guarda datos, pero tampoco da un error claro

Ir a Supabase → Authentication → Policies. Si la tabla tiene RLS activado sin políticas definidas, bloquea todo por defecto.

Para diagnosticar: desactivar RLS temporalmente ("Disable RLS" en la tabla) y verificar si los datos aparecen. Si sí, el problema está en las políticas.

Para apps sin autenticación de usuario: crear una política con USING (true) para permitir lectura pública. Para escritura, WITH CHECK (true).

Prompt útil: "Mi app no tiene autenticación de usuarios. Necesito que cualquier persona pueda leer y escribir en la tabla [nombre]. ¿Qué política de RLS necesito en Supabase?"

📝
Formulario sin validación
Se puede enviar vacío o con datos inválidos sin ningún mensaje de error

Primer paso: atributo required en los campos del HTML. Rápido y funciona en todos los navegadores.

Mensajes custom: manejar el estado de error en el componente y mostrar texto explicativo debajo del campo afectado. No solo un borde rojo — texto que dice qué está mal.

Prompt: "Agregar validación al formulario. Los campos [lista] son requeridos. Si el usuario intenta enviar sin completarlos, mostrar un mensaje claro debajo de cada campo vacío. Deshabilitar el botón de envío hasta que los campos obligatorios estén completos."

📭
Estado vacío sin manejar
Lista vacía sin contexto: el usuario no sabe si algo falló o si simplemente no hay datos

Condicionar el render: si el array tiene 0 elementos, mostrar un empty state en lugar de la lista vacía.

El empty state tiene tres partes: un ícono o ilustración, una línea de texto explicativa en el vocabulario del dominio, y un botón de call to action que lleva a crear el primer elemento.

Prompt: "Cuando la lista de [entidad] esté vacía, mostrar un estado vacío con el ícono [emoji], el mensaje '[texto relevante del dominio]' y un botón que abre el formulario para crear el primer [entidad]."

⏳
Loading states ausentes
El usuario hace click y no pasa nada visible por 2-3 segundos — puede pensar que se rompió

Deshabilitar el botón durante la operación y cambiar su texto. Es la implementación mínima viable de un loading state.

Para operaciones más largas: agregar un spinner o barra de progreso.

Prompt: "Mientras se procesa el submit del formulario, deshabilitar el botón y cambiar su texto a 'Guardando...' con un spinner. Cuando la operación termina (éxito o error), restaurar el botón a su estado normal."

📱
Layout roto en pantallas angostas
Elementos superpuestos, texto desbordando, botones cortados o desaparecidos en mobile

Revisar: ¿hay widths fijos que deberían ser max-width? ¿Flexbox sin flex-wrap? ¿Grid con columnas fijas que deberían ser auto-fit?

Prompt: "Hacer el layout de [nombre de la vista] completamente responsive. En pantallas menores a 640px, apilar los elementos verticalmente. Asegurar que ningún texto ni elemento desborde el viewport. Los botones deben tener al menos 44px de altura para ser tocables en mobile."

💬
Texto genérico de v0 sin personalizar
Placeholders "Enter text here", títulos "My App", botones "Submit" o "Click here"

Cada texto genérico rompe la ilusión de producto real. El copy es parte del diseño — dice quién sos y si entendés el problema del usuario.

Listar todos los textos de la interfaz y reemplazarlos por vocabulario específico del dominio.

Prompt: "Revisar todos los textos, labels, placeholders y mensajes de la interfaz. Reemplazar cualquier texto genérico por texto específico del dominio [nombre del dominio], usando el vocabulario que usaría el usuario real. El tono debe ser [formal/informal/etc.]."

"El criterio de éxito del build no es que el código se vea prolijo. Es poder hacer el flujo completo en incógnito, de principio a fin, sin errores, sin explicaciones."

Bloque 3

Demo Day

Dos minutos por persona. No es una presentación técnica — es una demostración de valor. El objetivo es que el que escucha entienda para qué sirve, quién lo usa y por qué importa, sin que menciones una sola línea de código.

Las tres reglas del demo

  • ✗
    No explicar el código. "Lo que hice fue conectar la tabla de Supabase con el componente React y..." — nadie que no sea desarrollador sabe qué significa eso. Hablá del problema y la solución desde el punto de vista del usuario.
  • ✗
    No decir "IA", "modelo", "API", "base de datos", "deploy". Estas palabras son invisibles para un cliente real. Usá el vocabulario del dominio — si es una app para turnos médicos, hablá de turnos, pacientes, consultorios, no de endpoints y webhooks.
  • ✗
    No pedir disculpas por lo que falta. No arranques diciendo "todavía me falta..." o "sé que no está terminado pero...". Mostrá lo que funciona con confianza. Los próximos pasos van al final, como hoja de ruta, no como disculpa.
  • ✓
    Aplicar la estructura consultiva del Módulo 7. Quién tiene el problema → cuál es el problema concreto → qué hace tu solución → cuál es el beneficio específico y medible. Después, demo en vivo de la URL.

Estructura de los 2 minutos

30s
Problema + usuario

¿Quién tiene el problema? ¿Qué problema concreto tienen? ¿Por qué importa resolverlo?

"Los dueños de veterinarias chicas pasan 90 minutos por día respondiendo las mismas preguntas de WhatsApp. Mi app responde esas preguntas automáticamente."

90s
Demo en vivo

Compartir pantalla con la URL de Vercel (o localhost:3000) abierta en incógnito. Hacer el flujo principal completo — el que muestra el valor central, no un tour de todas las pantallas.

Mostrar el flujo que resuelve el problema que describiste en los 30 segundos previos.

Las tres preguntas de feedback del grupo

"¿Entendiste para qué sirve?"
Evalúa claridad comunicacional. Si la mayoría no entendió, el problema de comunicación es tan importante como cualquier bug técnico.
"¿Lo usarías si tuvieras ese problema?"
Evalúa si la propuesta de valor es real. Respuesta honesta, no de cortesía. "Capaz" también es información útil.
"¿Qué le agregarías para que sea completamente real?"
Una sola cosa. No una lista. La más importante. Eso revela cuál es el gap más grande desde la perspectiva de alguien externo.

"El cliente no elige solo una solución: elige a la persona que lo va a acompañar en un proceso de cambio. El demo no es de código — es de confianza."

Cierre

Próximos pasos a producción

Lo que construiste en estas dos clases es un MVP. No es producción todavía. Para que sea un producto real que pueda usar un cliente que pagó por él, hay piezas que faltan. Identificalas, priorizalas, y empezá por las más críticas para tu caso específico.

🔑
Autenticación de usuarios
Si la app maneja datos de usuarios o clientes, necesitás Supabase Auth o equivalente. Sin autenticación, cualquier persona puede ver o modificar todos los datos.
Crítico si hay datos privados
🌐
Dominio propio
La URL de Vercel sirve para demos. Un cliente real necesita ver un dominio propio — dice que tomás el proyecto en serio. Registrar en Namecheap, Porkbun o similar y conectarlo a Vercel.
Antes del primer cliente
📊
Plan pago de Supabase
El plan gratuito de Supabase pausa los proyectos después de 7 días de inactividad y tiene límites de filas y storage. Si la app va a tener usuarios reales con tráfico, upgradear antes de que lleguen.
Cuando haya tráfico real
👥
Testing con usuarios reales
No con vos, no con compañeros del curso — con 3 a 5 personas que tengan el problema que la app resuelve y que nunca hayan visto el producto. Observar sin intervenir. Documentar dónde se traban.
Lo antes posible
📄
Documentación básica
Un README que explica qué hace la app, cómo se usa, qué datos guarda y cómo se despliega. Si alguien más tiene que mantenerla en 6 meses, ¿puede entenderla sin preguntarte?
Antes de pasar el proyecto
🔒
Políticas de RLS en Supabase
Si desactivaste RLS para debugging, volvelo a activar con las políticas correctas antes de tener usuarios reales. Acceso sin restricciones a la base de datos no es una opción en producción.
Antes de producción
🎯
Primer cliente potencial
Identificar una persona real con el problema que resuelve la app. Mostrarle la demo. Preguntar honestamente: "¿Lo usarías? ¿Qué le falta para que lo uses?" Ese feedback vale más que 10 features nuevas.
Esta semana
📈
Métricas básicas
¿Cómo vas a saber si el producto está resolviendo el problema? Definir una métrica de éxito concreta antes de tener usuarios — no cantidad de visitas, sino el indicador que importa para el caso de uso.
Con usuarios activos

"Construyeron algo que no existía hace dos semanas. La pregunta ya no es si podés hacerlo — es qué vas a hacer con eso."

Ejercicio de cierre: escribí tres próximos pasos concretos para tu proyecto específico. No genéricos — con nombre, acción y fecha. "Conseguir 5 personas del rubro para que prueben la app antes del [fecha]." Ese nivel de especificidad es el que produce resultados.
